\chapter{Proposed System Requirements}

This chapter describes the proposed Jan-Sunwai AI solution architecture from a requirement realization perspective. While Chapter 3 documented what is required, this chapter explains how the proposed system addresses those requirements through modular capabilities and integrated workflows built on the React, FastAPI, MongoDB, and Ollama stack [12, 11, 13, 17].

\section{Overview of Proposed System}

The proposed system is a role-driven civic complaint platform that combines AI-assisted intake with deterministic governance controls. Rather than adopting a monolithic architecture or a purely generative AI approach, the system was designed with strict module boundaries, a hybrid inference pipeline, and a flexible document-store database. These architectural choices were made to ensure the platform could run efficiently in resource-constrained municipal environments while maintaining strict auditability and data residency [11, 13, 7].

The system can be summarized as a five-stage path:

\begin{enumerate}
\item Complaint Intake: Citizen uploads image and requests analysis.
\item AI Enrichment: System extracts likely category, location hints, and draft text.
\item Structured Submission: Citizen reviews/edits and confirms complaint payload.
\item Operational Routing: Complaint is assigned, tracked, and escalated if needed.
\item Governance and Transparency: Notifications, analytics, exports, and public feed support accountability.
\end{enumerate}

\section{Module Descriptions}

The proposed system is organized as cooperating modules so that each operational responsibility remains isolated, testable, and maintainable. This modular decomposition also supports phased enhancement, where future upgrades can be applied to specific capability areas without destabilizing the entire platform. The subsections below summarize each major module and the role it plays in requirement realization.

\subsection{Authentication and User Management Module}

This module handles registration, login, session identity, profile updates, and password recovery. It enforces role-oriented authorization for citizen, worker, department head, and admin personas. Key controls include secure cookie-session issuance [10], role checks, JWT compatibility for API clients, and reset-token hashing.

\subsection{Analyze and Complaint Intake Module}

This module processes uploaded images, invokes the hybrid classifier pipeline, extracts location metadata, and orchestrates draft generation. It supports both immediate and queued generation paths and returns timing metadata for observability [3, 4, 5, 17].

\subsection{Complaint Lifecycle Module}

This module persists complaints and manages state transitions, including status updates, transfer, escalation, feedback, notes, and comments. It appends status history entries for each transition to preserve decision traceability.

\subsection{Assignment and Worker Operations Module}

This module auto-assigns complaints to eligible workers using department matching, optional geospatial service-area checks, and load balancing by active tasks. It also provides worker status controls and completion signaling [11, 13].

\subsection{Notification and Communication Module}

This module creates in-app notifications for complaint lifecycle events and maintains unread counters. It includes an email relay stub for deployment environments where SMTP integration is required.

\subsection{Triage and Governance Module}

This module supports admin review of low-confidence classifications. It enables decision actions such as approve, reject, or relabel. The triage path acts as a governance safety net against AI uncertainty [6, 5].

\subsection{Analytics and Public Transparency Module}

This module aggregates operational statistics, trend lines, and geospatial heatmap data. It also exposes an anonymized public feed to support transparency without leaking sensitive user data.

\section{Advantages of Proposed System}

The proposed solution offers measurable value across three levels: citizen experience, administrative throughput, and institutional governance. Crucially, the chosen architecture (FastAPI + React + MongoDB + Local LLM) outperforms simpler monolithic alternatives in several ways. The document model (MongoDB) is better suited to civic records, which evolve through complex, schema-less state transitions (such as appending notes and timestamps) without requiring expensive relational joins. Furthermore, the hybrid AI approach ensures that the non-deterministic nature of vision models is bounded by hard-coded, deterministic routing SLAs, making the system vastly more reliable for public service than a pure "black box" generative classifier.

\subsection{Citizen-Centric Advantages}

The citizen-facing advantages below stem directly from the image-first intake and AI-assisted drafting architecture:


\begin{itemize}
\item Simplified complaint filing with image-based assistance.
\item Better complaint quality due to generated formal drafts.
\item Real-time feedback through status and notification channels.
\item Reduced confusion in choosing the correct municipal department.
\end{itemize}

\subsection{Administrative and Departmental Advantages}

On the administrative side, the module boundaries and assignment automation translate into measurable operational gains:


\begin{itemize}
\item Better routing quality and lower manual triage burden for standard cases.
\item Strong lifecycle auditability for supervisory and escalation actions.
\item Faster assignment and backlog control through automation and bulk operations.
\item Structured data support for reporting and policy-level analysis.
\end{itemize}

\subsection{Institutional Advantages}

At the institutional level, the deployment architecture delivers compliance and handover advantages not possible with ad-hoc cloud-dependent designs:


\begin{itemize}
\item Local-first inference supports stricter data residency requirements.
\item Containerized deployment supports repeatable setup and handover.
\item Explicit API version aliasing improves integration stability.
\item Security hardening and validation controls reduce common exploit surfaces.
\end{itemize}

\section{Expected Impact}

The platform is expected to produce measurable improvements across six dimensions. Complaint quality will improve as guided image intake and AI drafting reduce inconsistent descriptions. Routing efficiency will increase as department prediction and authority mapping eliminate the first-hop re-assignment delay. Resolution governance becomes stronger through status timelines, notes, and escalation controls. Citizen trust improves through notifications and transparent progress signals. Operational analytics give administrators better planning tools through department trends and heatmap indicators. Finally, deployment readiness is lowered through a unified compose and environment strategy [1, 2, 6].

\section{Requirement Realization Detail by Persona}

Each persona receives a targeted feature set. \textbf{Citizens} get image-based intake, dashboard tracking, notifications, feedback, and password reset: reducing filing friction and improving visibility. \textbf{Workers} receive an assignment list, status update controls, and completion actions: delivering structured field execution. \textbf{Department Heads} have department-level filtering, status updates, transfer/escalation, and notes: enabling controlled supervisory governance. \textbf{Administrators} hold full operational governance: worker approvals, triage decisions, bulk operations, analytics, and exports. \textbf{Public Viewers} can access the anonymised complaint board without exposing any personally identifiable records.

\section{Proposed Data and Control Flow Narrative}

The proposed flow intentionally separates decision support from governance authority. AI modules suggest categories and draft text, but acceptance and lifecycle transitions remain controlled through authenticated role actions. This hybrid control plane avoids over-automation risk while still reducing manual burden in high-volume intake scenarios [5, 6, 11].

\section{Chapter Summary}

The proposed system translates requirement intent into integrated modules that jointly address user friction, operational traceability, and governance accountability. It deliberately combines AI assistance with deterministic controls and administrative safety nets, making it suitable for practical municipal deployment contexts.
