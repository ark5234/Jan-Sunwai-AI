\chapter{Project Management}

Project management for Jan-Sunwai AI followed an iterative delivery approach with explicit weekly milestones, acceptance checkpoints, and operational hardening gates. The planning strategy balanced feature development with risk reduction, especially in AI reliability, security posture, and deployment reproducibility.


\section{Feasibility Study}

The proposed system was evaluated for practical execution viability before and during implementation. Feasibility was treated as a living checkpoint rather than a one-time document, because model behavior, deployment constraints, and user workflow complexity evolved across iterations.

\subsection{Technical Feasibility}

Technical feasibility was evaluated through progressive integration rather than theoretical assessment. The React frontend, FastAPI backend, MongoDB store, Ollama runtime, and Docker Compose orchestration were brought into functional contact in the first two weeks of the project. Health endpoints were established and validated for each service dependency, and setup scripts were verified for idempotent execution on both Windows and Linux hosts. The primary technical risk identified during this phase was VRAM contention under concurrent inference requests, which informed the subsequent decision to introduce an asynchronous generation queue rather than processing all AI requests synchronously.

\subsection{Time Schedule Feasibility}

Time feasibility was assessed by decomposing the full feature scope into four phases: foundation (authentication and core complaint lifecycle), feature development (AI pipeline and role dashboards), security and reliability hardening, and documentation and delivery preparation. An eighteen-week window was mapped to this decomposition with explicit milestone dates and acceptance criteria at the boundary of each phase. The schedule was reviewed weekly against actual delivery progress, and scope was managed to ensure that hardening and documentation phases were not compressed to accommodate late-stage feature additions.

\subsection{Operational Feasibility}

Operational feasibility was confirmed by the construction of role-based workflows for all four user personas. Each persona was mapped to a distinct set of interactions, and those interactions were validated end-to-end through manual UAT scripts before the phase in which they appeared was considered closed. Feasibility at the operational level required that a citizen with limited digital literacy could navigate from image upload to confirmed submission in under five minutes, and that an administrator could perform a triage decision in under two minutes per queued item.

\subsection{Implementation Feasibility}

Implementation feasibility was substantiated through the successful execution of containerised setup scripts, environment templates, and validation routines across both development and production-profile Docker Compose configurations. The ability to reproduce a working deployment from a clean machine using a single command sequence was treated as the acceptance criterion for this feasibility dimension.



\section{Project Planning}

Project planning translated broad objectives into actionable delivery slices with explicit milestones, quality gates, and integration checkpoints. Planning was intentionally cross-functional, so backend APIs, frontend role flows, AI services, and deployment scripts evolved in synchronized increments rather than in isolated tracks. This structure reduced integration shocks and enabled predictable weekly review cycles.

\subsection{Development Approach \& Justification}

A sequential waterfall model was not a good fit for this project because the AI layer and the role-based user flows both needed repeated adjustment as implementation details became visible \cite{Docker2026, FastAPI2026, MongoDB2026}. The work therefore moved forward in short verified increments. Each increment began with a working baseline, moved through unit and integration checks, surfaced the relevant role workflow in the interface for end-to-end validation, and then received security and performance hardening once the core behaviour had stabilised. Release notes and operational steps were only written after those checks passed, so documentation reflected the version that was actually ready for handover.


\subsection{Milestones \& Deliverables}

Milestones and deliverables were tracked in weekly slices aligned to project governance checkpoints.

\begin{itemize}
\item Foundation phase: authentication, baseline complaint lifecycle, and core schema validation.
\item Feature phase: AI-assisted analysis, worker assignment, role dashboards, and notifications.
\item Hardening phase: upload security, resilience handling, and load/security verification suites.
\item Delivery phase: deployment validation, final documentation, and report artifacts.
\end{itemize}

\subsection{Group Dependencies}

The project includes cross-cutting dependencies that were managed explicitly:

\begin{itemize}
\item Backend depends on MongoDB availability and correctly configured environment variables.
\item Analyze flow depends on file storage service and Ollama runtime health.
\item Frontend route behavior depends on cookie-session state and role claims.
\item Assignment quality depends on worker profile completeness (department, approval, service area, status).
\item Production readiness depends on compose health checks, proxy alignment, and secure secret configuration.
\end{itemize}

Dependency risks were controlled through health endpoints, fallback behaviors, and scripted verification routines.

\section{Management Outcomes}

The iterative delivery strategy achieved three outcomes of note. Full-scope feature delivery was completed without deferring security or reliability hardening to a post-feature phase, which is a common failure mode in project timelines where feature work expands to fill available time. Delivery confidence was sustained by continuous automated testing and rapid issue-resolution cycles following each integration point. Handover readiness was advanced through consolidated deployment artefacts, environment documentation, and an API reference produced during the final two weeks rather than assembled retrospectively.



\section{Chapter Summary}

This chapter demonstrated that the project was not only technically implementable but also deliverable within a controlled timeline and governance framework. Feasibility, dependency management, risk planning, and milestone tracking supported predictable progress toward a production-oriented civic platform.
