\chapter{Project Management}

The project management for Jan-Sunwai AI was iterative; with weekly action milestones, acceptance checkpoints, and operational hardening gates. The planning strategy balanced feature development against risk minimization, particularly in AI reliability, security posture, and deployment reproducibility.

\section{Feasibility Study}

The proposed system was evaluated for practical execution viability at the project's outset and at multiple points during implementation. Feasibility was treated as a dynamic checkpoint rather than a static document, as model behaviour, deployment constraints, and user workflow complexity evolved throughout the project lifecycle.

\subsection{Technical Feasibility}

Technical feasibility was evaluated through progressive integration rather than purely theoretical assessment. The React frontend, FastAPI backend, MongoDB database, Ollama runtime, and Docker Compose orchestration were brought into functional contact during the first two weeks. Health endpoints were established and validated for each service dependency, and setup scripts were verified for idempotent execution on both Windows and Linux hosts. VRAM contention during concurrent inference requests emerged as the primary technical risk, leading to the introduction of an asynchronous generation queue in place of synchronous AI processing.

\subsection{Time Schedule Feasibility}

Time feasibility was assessed by decomposing the full feature scope into four sequential phases: foundation (authentication and core complaint lifecycle), feature development (AI pipeline and role dashboards), security and reliability hardening, and documentation and delivery preparation. An 18-week project window was allocated to this decomposition with explicit milestone dates and acceptance criteria at each phase boundary. Weekly progress reviews measured actual delivery against schedule, and scope was actively managed to prevent the hardening and documentation phases from being compressed by late-stage feature additions.

\subsection{Operational Feasibility}

Operational feasibility was validated through the construction of role-based workflows for all four user personas. Each persona was mapped to a specific set of interactions, and those interactions were validated end-to-end via manual UAT scripts before the phase containing them was considered closed. At the operational level, feasibility required that a citizen with limited digital literacy could complete the upload-to-submission flow in under five minutes, and that an administrator could complete a triage decision in under two minutes per queued item.

\subsection{Implementation Feasibility}

Implementation feasibility was demonstrated through successful execution of containerised setup scripts, environment templates, and validation routines across both development and production-profile Docker Compose configurations. The ability to reproduce a working deployment from a clean machine using a single command sequence was established as the acceptance criterion for this feasibility dimension.



\section{Project Planning}

Project planning converted broad objectives into actionable delivery slices with explicit milestones, quality gates, and integration checkpoints. Planning was intentionally cross-functional, ensuring that backend APIs, frontend role flows, AI services, and deployment scripts evolved in synchronized increments rather than in isolated work streams. This structure reduced integration surprises and enabled predictable weekly review cycles.

\subsection{Development Approach and Justification}

A sequential waterfall model was not appropriate for this project because the AI layer and role-based user workflows both required repeated refinement as implementation details emerged. The work therefore proceeded in short verified increments following agile iterative principles~\cite{agile_manifesto}. Each increment began with a working baseline, moved through unit and integration checks, surfaced the relevant role workflow in the interface for end-to-end validation, and received security and performance hardening once the core behaviour had stabilised. Release notes and operational procedures were written only after verification was complete, ensuring documentation reflected the version actually ready for deployment.


\subsection{Milestones and Deliverables}

Milestones and deliverables were tracked in weekly slices aligned to project governance checkpoints:

\begin{itemize}
\item Foundation phase: authentication, baseline complaint lifecycle, core schema validation
\item Feature phase: AI-assisted analysis, worker assignment, role dashboards, notifications
\item Hardening phase: upload security, resilience handling, load and security verification
\item Delivery phase: deployment validation, final documentation, report compilation
\end{itemize}

\subsection{Group Dependencies}

The project includes cross-cutting dependencies that were managed explicitly:

\begin{itemize}
\item Backend depends on MongoDB availability and correctly configured environment variables
\item Analyse flow depends on file storage service and Ollama runtime health
\item Frontend route behaviour depends on cookie-session state and role claims
\item Assignment quality depends on worker profile completeness (department, approval, service area, status)
\item Production readiness depends on compose health checks, proxy alignment, secure secret configuration
\end{itemize}

Dependency risks were controlled through health endpoints, fallback mechanisms, and scripted verification routines.

\section{Management Outcomes}

The iterative delivery strategy achieved three outcomes of note. Full-scope feature delivery was completed without deferring security or reliability hardening to a post-feature phase, which is a common failure mode in project timelines where feature work expands to fill available time. Delivery confidence was sustained by continuous automated testing and rapid issue-resolution cycles following each integration point. Handover readiness was advanced through consolidated deployment artefacts, environment documentation, and an API reference produced during the final two weeks rather than assembled retrospectively.



\section{Chapter Summary}

This chapter demonstrated that the project was not only technically implementable but also deliverable within a controlled timeline and governance framework. Feasibility, dependency management, risk planning, and milestone tracking supported predictable progress toward a production-oriented civic platform.
