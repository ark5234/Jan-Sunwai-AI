\chapter{System Requirement Study}

This chapter documents requirement analysis for Jan-Sunwai AI, including the baseline state of existing grievance systems, observed limitations, actor expectations, and formalized functional/non-functional requirements. The study was derived from implementation realities, municipal workflow assumptions, and validation through tests and user journeys.

\section{Existing System Overview}

Many conventional civic complaint channels depend on one or more of the following interfaces: physical counters, basic web forms, call center logging, or generic ticketing portals. While these methods can register complaints, they often suffer from weak input quality, manual routing dependency, and low traceability.

Key characteristics of existing systems observed in this problem context:

\begin{itemize}
\item Citizens are expected to manually choose departments, often without sufficient guidance.
\item Complaint narratives are text-heavy and unstructured, leading to inconsistent detail quality.
\item Photo evidence may be accepted but is rarely integrated into routing logic.
\item Status updates are frequently sparse or delayed.
\item Workflow metadata (who changed what and why) is often incomplete for audits.
\end{itemize}

\section{Constraints}

\subsection{UI Constraints}

\begin{itemize}
\item Role-specific pages must enforce access boundaries and clear redirection behavior.
\item Analyze/result screens must preserve draft continuity during refresh and error conditions.
\item Dashboard cards and maps must remain usable at mobile widths (e.g., 375 px baseline tests).
\end{itemize}

\subsection{Communication Interface}

\begin{itemize}
\item REST over HTTP(S) is the canonical communication model.
\item JWT bearer token is mandatory for protected routes.
\item Production frontend proxy must route API traffic to backend service with /api/v1 compatibility.
\end{itemize}

\subsection{Hardware Interface}

\begin{itemize}
\item AI response times depend on model runtime health and available VRAM.
\item GPU contention and model load/unload latency can affect throughput.
\item Large image uploads require strict file-size and decode constraints to avoid resource exhaustion.
\end{itemize}

\subsection{Criticality of Application}

The system is governance-sensitive. Incorrect status transitions, misrouted complaints, or unauthorized updates can directly impact citizen trust and administrative accountability. Hence, status history and role checks are non-negotiable controls.

\subsection{Safety and Security Considerations}

\begin{itemize}
\item Input sanitization is required for free-text fields.
\item File intake must detect extension-header mismatches and corrupted payloads.
\item Token reset flow must avoid account enumeration and raw token persistence.
\item Security headers and CORS policies must remain active on both normal and error responses.
\end{itemize}

\section{Chapter Summary}

The requirement study establishes the transition from a manually intensive grievance flow to a structured, secure, and auditable digital workflow. Functional and non-functional requirements were formalized in implementation-aligned terms and mapped against realistic user roles and operational constraints.
