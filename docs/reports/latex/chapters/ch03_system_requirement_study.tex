\chapter{System Requirement Study}

This chapter documents the requirements analysis for Jan-Sunwai AI, including the baseline state of existing grievance systems, observed limitations, actor expectations, and formalised functional and non-functional requirements. The study was grounded in implementation practicalities, municipal workflow assumptions, and validation through testing and user journeys.

\section{Existing System Overview}

Most conventional civic complaint channels operate through one or more interfaces: physical counters, basic web forms, call centre logging, or generic ticketing portals. While these methods can register complaints, they often suffer from weak input quality, manual routing dependency, and inadequate traceability.

Key characteristics of existing systems in this problem domain include:

\begin{itemize}
\item Citizens manually select departments, frequently without adequate guidance
\item Complaint narratives are text-heavy and unstructured, resulting in inconsistent detail quality
\item Photographic evidence may be accepted but is rarely used in routing decisions
\item Status updates are often sparse or delayed
\item Workflow metadata (actor attribution, change timestamps) is often incomplete for audit purposes
\end{itemize}

\section{Limitations of the Existing System}

The operational deficiencies in legacy civic complaint channels fall into several broad categories. Citizen-initiated department selection is the most significant structural weakness because it places classification responsibility on the population least equipped to exercise it correctly. Complaints submitted to the wrong department must be transferred before action is possible, and each transfer consumes a full administrative review cycle. Complaint narrative quality is uneven because unstructured free-text fields produce descriptions ranging from detailed to sparse, requiring staff to seek clarification before field action can be authorised. Photographic evidence, where accepted, is stored as an attachment rather than processed as routing input, meaning the most objective available record is never used for classification. Role-specific interface design is typically absent, so field workers, department supervisors, and administrators interact with identical interfaces despite having fundamentally different information needs. Lifecycle traceability is insufficient for governance, as state transitions without actor attribution and timestamps cannot support escalation justification or post-resolution review. Public visibility of queue behaviour is negligible, eroding citizen trust by creating the impression that complaints disappear without acknowledgement.

\section{User Characteristics}

Four distinct operational personas were identified during requirements analysis, each with characteristic interaction patterns and clearly defined authorisation boundaries.

Citizens constitute the largest user population. They interact with the platform at two main points: submission and tracking. During submission, a citizen uploads photographic evidence, reviews the AI-generated draft complaint, optionally modifies the description or category, and confirms submission. During tracking, the citizen monitors status updates through the notification panel and complaint detail view.

Field workers interact primarily through their assigned-task queue. They manage their own availability status, review complaint details and location information, and mark tasks as completed with a resolution note upon successful intervention.

Department heads exercise supervisory authority over their department's complaint queue. They apply status updates with explanatory notes, transfer complaints to other departments when jurisdictional analysis suggests mis-assignment, and escalate cases to parent authorities when the matter exceeds departmental capacity or authority.

Administrators hold end-to-end governance authority over the platform. Their responsibilities include approving or rejecting worker registration applications, adjudicating the triage queue of low-confidence classifications, executing bulk status operations, exporting complaint datasets as CSV files, and monitoring system-wide analytics.

\section{Hardware and Software Requirements}

A minimum hardware configuration of a four-core processor (eight-core recommended), 12 to 16 GB of system RAM, 4 to 6 GB of GPU VRAM, 10 to 20 GB of storage, and a stable network connection is required for functional operation. The software baseline requires Windows 10 or later, or Ubuntu 22.04 LTS or later, alongside Python 3.10 or later, Node.js v20 LTS or later, MongoDB 7.x, the Ollama runtime, Docker and Docker Compose version 24 or later, and a modern web browser.

\section{Constraints}

Constraints are not merely technical limits but also governance and usability boundaries that directly influence system behaviour. The platform must operate under security, role-separation, and infrastructure limitations common in civic environments while maintaining user clarity for citizens and workflow efficiency for officials. The following constraint groups define implementation boundaries and informed several architecture and UI decisions.

\subsection{User Interface Constraints}

Role-specific pages enforce access boundaries. Interfaces support mobile widths (375px baseline) and preserve data state during refresh cycles so users can recover context after reconnection.

\subsection{Communication Interface}

Communication occurs exclusively over authenticated REST endpoints via secure cookie sessions or JWT tokens, with strict /api/v1 proxy routing and controlled CORS policy.

\subsection{Hardware Interface}

As a closed-system deployment, local inference is bound by available VRAM, causing AI response times to fluctuate under load. Large file uploads are strictly limited by size and decode verification to prevent resource exhaustion.

\subsection{Application Criticality}

The application is operationally critical because it manages public grievance routing and status accountability. Any prolonged outage can delay municipal field response and reduce citizen trust in service delivery.

\subsection{Safety and Security Considerations}

To guarantee citizen trust and data privacy, the system runs on secure local infrastructure. Status history and role checks are immutable. Input sanitisation, MIME-type validation, robust token lifecycle controls, and security headers enforce defence-in-depth.

\section{Assumptions and Dependencies}

The system operates under several core assumptions and external dependencies. First, it assumes a reachable and healthy MongoDB instance for all persistence. Second, the Ollama runtime must be correctly configured with the required vision and reasoning models to support analysis and draft generation. Third, accurate environment variable configuration is essential for stable authentication, CORS, and database mapping. Fourth, the system relies on worker profile completeness (department and service-area metadata) for the auto-assignment engine. Finally, for production deployments, the host must support Docker and Docker Compose to ensure environment reproducibility.

Requirements are classified using the MoSCoW methodology. Must-Have features include user registration, authentication, complaint intake, status updates, auto-assignment, and health monitoring; these are essential for operational viability. Should-Have features include authority escalation, worker status controls, administrative triage queue, and analytics; these provide institutional-grade governance. Could-Have features such as CSV exports and the public transparency feed provide high-value reporting without blocking core workflow. Non-Functional criticals including session security, sanitisation, and observability are treated as mandatory baseline controls.

\section{Chapter Summary}

The requirement study establishes the transition from a manually intensive grievance flow to a structured, secure, and auditable digital workflow. Functional and non-functional requirements were formalised in implementation-aligned terms and mapped against realistic user roles and operational constraints.
