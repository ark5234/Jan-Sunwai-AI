\chapter{Implementation Planning}

Implementation planning for Jan-Sunwai AI focused on three parallel goals: reproducibility, maintainable module growth, and deployment readiness. The implementation strategy treated backend, frontend, AI runtime, and operational scripts as coordinated components rather than isolated code tracks.

\section{Implementation Environment}

\subsection{Local Development Environment}

The local workflow supports rapid iteration with role-based UI validation and backend testing. Typical development topology includes:

\begin{itemize}
\item Backend running on FastAPI/Uvicorn at port 8000.
\item Frontend development server on Vite at port 5173.
\item MongoDB either local service or containerized dependency.
\item Ollama runtime with configured model set on host machine.
\end{itemize}

The local setup was selected to reduce iteration latency while preserving fidelity with production behavior through shared APIs, environment variables, and health semantics.

\subsection{Production-Style Container Environment}

For deployment simulation and handover readiness, the project includes a unified compose configuration with a production profile. In production mode:

\begin{itemize}
\item MongoDB runs as a dedicated service with health checks.
\item Backend runs from production Dockerfile and mounts uploads volume to \texttt{/app/uploads}.
\item Frontend runs in Nginx container under compose profile \texttt{prod}.
\item Health checks and log rotation policies are explicitly configured.
\end{itemize}

This ensures that local verification and deployment handover use the same operational assumptions.

\section{Coding Standards Followed}

\subsection{Backend Standards}

The points below highlight the most important considerations that define this subsection.


\begin{itemize}
\item Router-based modularization for domain isolation.
\item Strict schema validation through typed request models.
\item Clear HTTP semantics for validation, auth, and lifecycle errors.
\item Service extraction for cross-cutting concerns (assignment, sanitization, storage, email).
\item Structured logging and failure-safe response behavior.
\end{itemize}

\subsection{Frontend Standards}

The points below highlight the most important considerations that define this subsection.


\begin{itemize}
\item Role-aware route guards and dashboard redirection.
\item Separation of pages, reusable components, layouts, and context providers.
\item Consistent API base usage through environment variables.
\item Explicit loading/error states for user-facing long-running operations.
\item Responsive design checks for desktop and mobile interaction baselines.
\end{itemize}

\subsection{Configuration Standards}

The points below highlight the most important considerations that define this subsection.


\begin{itemize}
\item Single canonical backend environment file for consistency.
\item Explicit production defaults and overrides for sensitive settings.
\item Avoidance of duplicate compose and env artifacts to reduce drift.
\item Documented scripts for startup, tests, deployment simulation, and verification.
\end{itemize}

\section{Implementation Work Packages}

\subsection{Work Package 1: Core Backend and Data Layer}

Deliverables included app bootstrap, database connection lifecycle, auth primitives, user and complaint schemas, and initial complaint CRUD operations. This package established foundational APIs and validation behavior.

\subsection{Work Package 2: AI Pipeline Integration}

Deliverables included classifier orchestration, model fallback strategy, rule-engine scoring, queue-based generation path, and timing metadata in analysis responses. This package transformed basic upload capability into intelligent intake flow.

\subsection{Work Package 3: Frontend Role Flows}

Deliverables included authentication screens, analyze/result pathway, role dashboards, notifications UI, and profile controls. Route guards and dashboard routing ensured role-appropriate user journeys.

\subsection{Work Package 4: Operational Features}

Deliverables included worker assignment logic, timeline/audit features, comments/notes, escalation, triage, analytics, and CSV exports. This package completed department-level and governance-level operational requirements.

\subsection{Work Package 5: Security and Reliability Hardening}

Deliverables included upload integrity checks, sanitization, security headers, CORS tightening, rate-limiting toggles, readiness/liveness checks, graceful AI degradation, and resilience-focused tests.

\subsection{Work Package 6: Deployment and Handover Readiness}

Deliverables included the production compose profile, deployment guides, and API references. The package also included backup scripts and verification procedures. This package enabled transition from a development artifact to a deployable product.

\section{Configuration and Secret Planning}

The implementation plan emphasized explicit secret handling and runtime configurability. Important control groups include:

\begin{itemize}
\item Authentication controls: JWT secret, algorithm, token expiry.
\item Data controls: Mongo URL and database selection.
\item AI controls: model identifiers, timeout windows, queue worker count.
\item Security controls: CORS origin allowlist and rate-limiter mode.
\item Communication controls: SMTP host/port/from for notification relay.
\end{itemize}

A notable planning correction during implementation was moving to one canonical backend environment file to eliminate drift between local and production templates.

\subsection{Rollback Planning}

Rollback planning was included to avoid high-impact downtime during deployment changes:

\begin{itemize}
\item Preserve previous container images and compose configuration snapshots.
\item Ensure database backup is taken before schema/index-affecting changes.
\item Revert frontend image and backend service independently when only one tier regresses.
\item Maintain known-good environment variable snapshot for rapid restoration.
\end{itemize}

\section{Change Management and Release Strategy}

\subsection{Release Checklist}

The sequence below presents the recommended order of actions and priorities for this subsection.


\begin{enumerate}
\item Confirm automated tests pass for backend and frontend quality gates.
\item Verify compose validation for default and production profiles.
\item Validate security controls: headers, CORS allowlist, upload checks, and role restrictions.
\item Confirm migration/index scripts produce expected database state.
\item Review documentation deltas (API, deployment, report references) before tagging release.
\item Capture release notes with risk statements and rollback pointers.
\end{enumerate}

\subsection{Implementation Learnings}

Implementation planning revealed that operational simplification tasks (for example, consolidating environment and compose artifacts) can have impact comparable to feature work. These tasks reduced recurring integration errors, made CI behavior more predictable, and improved deployment communication across stakeholders.

\section{Chapter Summary}

Implementation planning translated conceptual design into execution-ready work packages, sequence constraints, and quality gates. The plan successfully balanced rapid feature delivery with operational robustness, resulting in a deployable and documentable civic platform.
